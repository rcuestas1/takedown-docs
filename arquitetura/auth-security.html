Autenticacao & Seguranca ​

Autenticacao via Cookies ​

O Kubex usa httpOnly cookies (nao localStorage) para autenticacao:

access_token  — JWT, expira em 15 minutos
refresh_token — JWT, expira em 7 dias

JWT Tokens ​

Todos os tokens JWT incluem:

json
{
  "sub": "user_id",
  "type": "access|refresh|reset|verify",
  "exp": 1234567890,
  "iat": 1234567890
}
TipoExpiracaoUso
access15 minAutenticacao de requests
refresh7 diasRenovacao do access_token
reset1 horaReset de senha
verify24 horasVerificacao de email

Algoritmo: HS256 com SECRET_KEY.

Hashing de Senhas ​

  • bcrypt com cost factor 12 (configuravel via BCRYPT_ROUNDS)
  • Execucao async via asyncio.to_thread para nao bloquear o event loop
  • Anti-timing-attack: hash dummy executado mesmo quando usuario nao existe
python
await hash_password(password)        # Retorna hash bcrypt
await verify_password(plain, hashed) # Retorna bool

2FA (TOTP) ​

Autenticacao de dois fatores via TOTP (Time-based One-Time Password):

  1. Setup: POST /auth/setup-2fa → retorna provisioning_uri + qr_code_base64
  2. Ativacao: POST /auth/verify-2fa → valida codigo e ativa 2FA
  3. Login: Inclui totp_code no login
  4. Desativar: POST /auth/disable-2fa → requer senha + codigo TOTP
  • Biblioteca: pyotp
  • valid_window=1 — aceita codigo anterior/proximo (tolerancia de 30s)
  • QR code gerado como PNG base64 via qrcode library
  • Issuer: "Kubex"

Criptografia de Dados Sensiveis ​

AES-256-GCM (CPF/CNPJ) ​

Dados sensíveis criptografados em repouso:

python
encrypt(plaintext) → "nonce_b64:ciphertext_b64"
decrypt(blob) → plaintext
  • Algoritmo: AES-256-GCM (autenticado)
  • Nonce: 96 bits aleatorios por criptografia
  • Chave: ENCRYPTION_KEY deve ter exatamente 64 caracteres hexadecimais (32 bytes). Sem fallback de derivacao
  • Singleton: Instancia AESGCM cacheada em memoria

HMAC-SHA256 (Unicidade) ​

Para verificar unicidade sem decifrar:

python
hmac_hash(cpf_or_cnpj) → hex string (64 chars)

Armazenado em cpf_hash / cnpj_hash com constraint UNIQUE.

bcrypt (Senhas) ​

  • Cost factor 12 (configuravel via BCRYPT_ROUNDS)
  • Execucao async via asyncio.to_thread
  • Anti-timing-attack: hash dummy quando usuario nao existe
python
await hash_password(password)        # Retorna hash bcrypt
await verify_password(plain, hashed) # Retorna bool

JWT (Tokens) ​

  • Algoritmo: HS256 com SECRET_KEY
  • Tipos: access (15min), refresh (7d), reset (1h), verify (24h)
  • Payload: sub (user_id), type, exp, iat

Protecao CSRF ​

Middleware ASGI que valida o header Origin em requests state-changing (POST, PUT, PATCH, DELETE):

  • Validacao: Compara Origin contra CORS_ORIGINS configurados
  • Endpoints isentos: /api/v1/payments/webhook, /api/health, /api/ready
  • Dev sem Origin: Permitido (ferramentas como curl/Postman)
  • Prod sem Origin: Rejeitado com 403 Forbidden
  • Arquivo: app/middleware/csrf.py

Stack de middlewares (ordem de execucao):

SecurityHeaders → Logging → CSRF → Maintenance → app

CORS ​

Configuracao restritiva, mesma origem via Nginx:

python
CORS_ORIGINS = [
    "http://kubex.test",
    "http://admin.kubex.test",
    "http://localhost:3000"
]
  • Credentials: true (necessario para cookies)
  • Methods: GET, POST, PATCH, DELETE, OPTIONS
  • Headers: Content-Type, Authorization

reCAPTCHA v3 ​

  • Verificado no registro de usuarios (POST /auth/register)
  • Score minimo: 0.5
  • Desabilitado em dev (quando RECAPTCHA_SECRET_KEY ausente)
  • Frontend exibe erro se reCAPTCHA falhar (nao envia request sem token em prod)

Protecao contra Enumeracao ​

  • Registro: Resposta generica mesmo se email ja existe
  • Forgot Password: Sempre retorna sucesso
  • Timing: Hash dummy executado para equalizar tempo de resposta

Validacao de Imagens ​

  • Verificacao de formato real via Pillow (nao Content-Type)
  • Protecao contra decompression bombs (max 8000x8000 px)
  • Formatos aceitos: JPEG, PNG, WebP, GIF
  • Tamanho maximo: 10MB (busca), 5MB (selfie)
