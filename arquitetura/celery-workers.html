Celery Workers ​

Arquitetura dos workers Celery do Kubex: engine de banco compartilhado, filas priorizadas, roteamento de tarefas e ciclo de vida dos processos.

Engine de Banco Compartilhado ​

Arquivo: app/tasks/db.py

Cada processo worker (prefork child) mantém um singleton de AsyncEngine do SQLAlchemy, inicializado preguicosamente apos o fork via sinal worker_process_init.

python
# Parametros do pool por worker process
pool_size=5
max_overflow=10
pool_recycle=1800   # 30 min — evita conexoes stale
pool_pre_ping=True  # valida conexao antes de usar

Por que singleton? ​

O padrao antigo criava um novo engine por tarefa executada. Sob carga, isso causava exaustao do pool de conexoes do PostgreSQL — cada tarefa abria conexoes que nunca eram reutilizadas. O singleton garante que todas as tarefas de um mesmo processo compartilhem o mesmo pool, respeitando os limites configurados.

Funcoes disponiveis ​

FuncaoRetornoUso
get_task_engine()AsyncEngineAcesso direto ao engine (raro)
get_task_session()async_sessionmaker[AsyncSession]Fabrica de sessoes para tarefas
dispose_task_engine()NoneCleanup no shutdown

Importante: Nunca chame get_task_engine() em tempo de import. O engine so deve ser criado APOS o fork do worker.

Filas e Priorizacao ​

O Kubex usa 3 filas principais + 1 dedicada para processamento facial:

Ordem de consumo ​

O worker principal consome na ordem listada: celery_high → celery → celery_low. Isso significa que buscas iniciadas pelo usuario (Lens e BrandWatch) tem prioridade sobre monitoramentos automaticos.

FilaPrioridadeTarefas
celery_highAltasearch.run_image_search, search.run_name_search
celeryNormalTarefas sem roteamento explicito
celery_lowBaixamonitoring.*, discovery.*
celery_faceDedicadaembedding.* (InsightFace AntelopeV2)

Roteamento de Tarefas ​

Configurado em app/tasks/__init__.py via task_routes:

python
celery_app.conf.task_routes = {
    "search.run_image_search": {"queue": "celery_high"},
    "search.run_name_search":  {"queue": "celery_high"},
    "embedding.*":             {"queue": "celery_face"},
    "monitoring.*":            {"queue": "celery_low"},
    "discovery.*":             {"queue": "celery_low"},
    "bulk_crawl.*":            {"queue": "celery_low"},
}

Tarefas periodicas do Beat schedule usam "options": {"queue": "celery_low"} para nao competir com buscas do usuario:

python
"trigger-crawler-scans": {
    "task": "monitoring.trigger_crawler_scans",
    "schedule": 3600.0,
    "options": {"queue": "celery_low"},
},

Ciclo de Vida do Worker ​

Sinais registrados ​

SinalHandlerAcao
worker_process_initinit_task_engine()Inicializa engine apos fork (seguro para prefork)
worker_shutdowncleanup_task_engine()Executa dispose_task_engine() via asyncio.run()

Configuracao Docker Compose ​

Worker principal (dev) ​

bash
celery -A app.tasks worker \
  --loglevel=info \
  --concurrency=2 \
  --pool=prefork \
  -Q celery_high,celery,celery_low \
  --soft-time-limit=300 \
  --time-limit=600

Worker principal (prod) ​

bash
celery -A app.tasks worker \
  --loglevel=warning \
  --concurrency=4 \
  --pool=prefork \
  -Q celery_high,celery,celery_low

Face worker (dedicado) ​

Worker separado com a fila celery_face. Usa uma imagem Docker com InsightFace AntelopeV2 pre-instalado (build mais pesado). Concurrency: 4.

Parametros importantes ​

ParametroValorDescricao
task_soft_time_limit300s (5 min)Tarefa recebe SoftTimeLimitExceeded
task_time_limit600s (10 min)Hard kill do processo
task_acks_lateTrueRe-entrega se worker morrer antes de ACK
worker_max_tasks_per_child200Previne memory leaks — recicla processo
result_expires3600s (1h)TTL dos resultados no Redis

Progresso em Tempo Real (Redis) ​

As tarefas de busca (search.run_image_search, search.run_name_search) reportam progresso em tempo real via Redis:

python
# Em search_tasks.py — escrita sincrona (worker Celery nao tem event loop)
_sync_redis().setex(f"job_progress:{public_id}", 300, json.dumps({
    "step": "Buscando nos motores de busca",
    "percent": 25,
    "step_index": 1,
    "total_steps": 5,
}))
AspectoDetalhe
Redis clientSync (redis.from_url), singleton cached em _redis_client
Chavejob_progress:{public_id}
TTL300s (5 min) — auto-cleanup
ConsumidorGET /v1/search/jobs/{job_id}/progress (async, le do Redis Cache)
FallbackSe Redis nao tem dados, retorna "Aguardando na fila" com posicao

Tracing ​

Todas as tarefas Celery usam TracedTask como base class, que propaga o trace_id via headers (nao kwargs). Isso permite rastrear uma requisicao HTTP ate a execucao da tarefa nos logs. Veja Observabilidade para detalhes.
