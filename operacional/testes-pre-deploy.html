Roteiro de Testes Manuais Pre-Deploy ​

Siga cada teste na ordem. Cada passo tem a acao exata e o resultado esperado. Marque [x] conforme concluir. Tempo total estimado: ~2 horas.

Preparacao ​

1. Subir o ambiente:

bash
docker compose up -d

2. Verificar containers:

bash
docker compose ps

Esperado: api, db, redis, redis-cache, nginx, frontend, frontend-admin, celery-worker, celery-beat, face-worker, etcd, minio, milvus — todos "Up" ou "healthy"

3. Verificar hosts (C:\Windows\System32\drivers\etc\hosts):

127.0.0.1 kubex.test
127.0.0.1 admin.kubex.test

4. Instalar dependencias do monorepo (se primeira vez):

bash
npm install -g pnpm
pnpm install

5. Rodar testes automatizados:

bash
docker compose exec -T api python -m pytest tests/ -v --tb=short

Esperado: Todos os testes passam (94 arquivos, 500+ testes)

6. Verificar distributed tracing:

bash
curl -s -I http://kubex.test/api/health | grep -i x-trace-id

Esperado: X-Trace-Id: <32 hex chars>

7. Verificar Redis separado (broker vs cache):

bash
docker compose exec redis redis-cli -a kubex_dev_pass ping
docker compose exec redis-cache redis-cli -a kubex_dev_pass ping

Esperado: Ambos retornam PONG

8. Testar backup/restore:

bash
./scripts/backup-db.sh /tmp
ls -la /tmp/kubex_*.sql.gz

Esperado: Arquivo .sql.gz criado com tamanho > 0

Ativacao dos Motores (Mock → Real) ​

Por padrao, motores sem API keys retornam dados mock/dummy. Siga este guia para ativar cada motor com dados reais em dev.

Passo 0: Verificar platform_settings ​

Todos os motores sao gateados pelo campo search_engines da tabela platform_settings. Para verificar quais estao habilitados:

bash
docker compose exec -T db psql -U kubex -d kubex -c \
  "SELECT search_engines FROM platform_settings LIMIT 1;"

Para habilitar todos de uma vez via API admin:

bash
curl -X PATCH http://admin.kubex.test/api/v1/admin/system/settings \
  -H "Cookie: access_token=<TOKEN_ADMIN>" \
  -H "Content-Type: application/json" \
  -d '{
    "search_engines": [
      "lens", "crawler", "fingerprint", "brandwatch", "telegram",
      "faceid", "takedown", "orchestrator", "discord", "watermark",
      "videoprint", "semanticai", "reddit", "alertsfeed", "x_monitor",
      "darkweb", "sourceintel", "instagram", "marketplace", "ocr",
      "deepguard", "facebook", "flickr", "autodiscovery", "shieldai",
      "nsfwguard", "sentinel"
    ]
  }'

Ou pelo painel admin: System > Configuracoes > search_engines.

Lens — Busca por Imagem (SerpAPI) ​

O que faz: Busca reversa de imagem via APIs externas. Mock mode: Retorna 6 URLs dummy quando nenhuma API key esta configurada.

Para ativar:

  1. Obter API key:

    • SerpAPI: Criar conta em https://serpapi.com → plano gratis = 100 buscas/mes

  2. Adicionar no .env:

    env
    SERPAPI_API_KEY=sua-chave-serpapi

  3. Reiniciar: docker compose restart api celery-worker

  4. Verificar que "lens" esta no search_engines

Testar: Fazer upload de imagem em Buscar. Resultados reais do Google aparecem ao inves dos 6 URLs mock.

Crawler — Web Crawler (Scrapling) ​

O que faz: Crawleia sites cadastrados buscando imagens similares via pHash. Mock mode: Nao tem mock — simplesmente nao roda se desabilitado ou sem targets.

Para ativar:

  1. Verificar que "crawler" esta no search_engines

  2. Criar pelo menos um WebCrawlerTarget no banco:

    bash
    docker compose exec -T db psql -U kubex -d kubex -c "
  INSERT INTO web_crawler_targets (name, search_url_template, is_active, scrape_method, created_at, updated_at)
  VALUES
    ('Google Images', 'https://www.google.com/search?tbm=isch&q={query}', true, 'basic', NOW(), NOW()),
    ('Bing Images', 'https://www.bing.com/images/search?q={query}', true, 'basic', NOW(), NOW());
"

    Ou pelo painel admin: Crawlers > Adicionar Target

  3. Garantir que existem ContentHash no banco (gerados automaticamente ao fazer upload de imagem)

  4. Configuracoes opcionais no .env:

    env
    # Proxies (recomendado para evitar bloqueios)
CRAWLER_PROXY_URLS=["http://user:pass@host:port"]
# Bypass Cloudflare (requer Camoufox)
CRAWLER_ENABLE_CLOUDFLARE_BYPASS=false

  5. Reiniciar: docker compose restart celery-worker celery-beat

Testar: No admin > Crawlers, clicar "Executar" em um target. Verificar logs:

bash
docker compose logs celery-worker --tail=30 | grep -i "crawler\|crawl"

Fingerprint — pHash (Hashing Perceptual) ​

O que faz: Gera hashes perceptuais (pHash + dHash + aHash) de imagens. Mock mode: Nao tem — sempre ativo. Funciona localmente sem API externa.

Nenhuma configuracao necessaria. Ja funciona ao fazer upload de imagem.

BrandWatch — Busca por Nome/Marca ​

O que faz: Busca variacoes de nomes/marcas em sites cadastrados. Mock mode: Retorna URLs dummy se nao tem SERPAPI key nem CrawlTargets.

Para ativar:

  1. Verificar que "brandwatch" esta no search_engines

  2. Opcao A — Usar SerpAPI (mesma key do Lens):

    env
    SERPAPI_API_KEY=sua-chave-serpapi

  3. Opcao B — Criar CrawlTarget (URL templates para busca por nome):

    bash
    docker compose exec -T db psql -U kubex -d kubex -c "
  INSERT INTO crawl_targets (name, search_url_template, is_active, created_at, updated_at)
  VALUES
    ('Google Web', 'https://www.google.com/search?q={query}', true, NOW(), NOW());
"

  4. Reiniciar: docker compose restart api celery-worker

Testar: Menu > Monitorar Nome, cadastrar um nome e iniciar busca.

Nenhuma configuracao necessaria. Funciona localmente.

VideoPrint — Video Fingerprint ​

O que faz: Extrai keyframes de video e gera hashes por frame. Mock mode: Nao tem — requer ffmpeg instalado no container.

Para ativar:

  1. Verificar que "videoprint" esta no search_engines
  2. Verificar que ffmpeg esta disponivel:
    bash
    docker compose exec api ffmpeg -version
  3. O container ja deve incluir ffmpeg no Dockerfile. Se nao, adicionar:
    dockerfile
    RUN apt-get update && apt-get install -y ffmpeg

Testar: Menu > Video, fazer upload de MP4 curto.

SemanticAI — CLIP (IA Semantica) ​

O que faz: Gera embeddings de imagem para busca semantica. Mock mode: Gera embeddings deterministicos se open_clip nao esta instalado.

Para ativar com modelo real:

  1. Verificar que "semanticai" esta no search_engines
  2. Verificar que open_clip esta instalado:
    bash
    docker compose exec api python -c "import open_clip; print('OK')"
  3. Primeiro uso baixa o modelo (~400MB). GPU (CUDA) recomendado para performance.

DarkWeb Monitor — Dark Web (Tor) ​

O que faz: Monitora sites .onion via proxy Tor. Mock mode: Retorna [] quando TOR_SOCKS5_PROXY ausente (DarkWebMockMonitor).

Para ativar:

  1. Adicionar servico Tor no docker-compose.yml (se nao existir):

    yaml
    tor-proxy:
  image: dperson/torproxy
  restart: unless-stopped
  ports:
    - "9050:9050"

  2. Adicionar no .env:

    env
    TOR_SOCKS5_PROXY=socks5://tor-proxy:9050

  3. Verificar que "darkweb" esta no search_engines

  4. Reiniciar: docker compose up -d tor-proxy && docker compose restart celery-worker celery-beat

Nota: Motor restrito ao plano Enterprise. Delay de 5-15s entre sites por anti-deteccao.

SourceIntel — Source Discovery (Cross-Reference) ​

Para ativar:

  3. Crawler deve estar ativo — SourceIntel se alimenta dos links extraidos pelo crawler

  4. Reiniciar: docker compose restart celery-worker celery-beat

Instagram Monitor — Instagram Monitor + Discovery ​

O que faz: Duas funcionalidades:

  1. Discovery (Beat 6d, cooldown 7d/user): Busca ativa de contas fake — pega nome do usuario, gera variacoes de username, busca no Instagram via TopSearchResults, compara foto de perfil via pHash. Max 100 perfis/user, savepoint por insert.
  2. Scan (3h): Monitora perfis ativos (manuais + auto-descobertos), baixa novos posts e compara via pHash. Hashes pre-agrupados por user_id para O(1) lookup.

Mock mode: Retorna [] quando INSTAGRAM_MONITOR_ENABLED=false (InstagramMockMonitor).

Para ativar:

  1. Adicionar no .env:

    env
    INSTAGRAM_MONITOR_ENABLED=true

  2. Instalar dependencia:

    bash
    docker compose exec -T api pip install instaloader

  3. Verificar que "instagram" esta no search_engines

  4. Rodar migration: docker compose exec -T api alembic upgrade head

  5. (Opcional) Adicionar perfis manuais via admin:

    bash
    curl -X POST http://admin.kubex.test/api/v1/admin/social/instagram/profiles \
  -H "Content-Type: application/json" \
  --cookie "access_token=..." \
  -d '{"username": "perfil_suspeito", "label": "Teste"}'

  6. Reiniciar: docker compose restart celery-worker celery-beat

Testar Discovery: O discovery roda a cada 6 dias (com cooldown de 7d por usuario), mas pode ser disparado manualmente:

bash
docker compose exec -T celery-worker celery -A app.tasks call monitoring.trigger_instagram_discovery

Testar Scan: Verificar alertas apos scan:

bash
docker compose exec -T db psql -U kubex -d kubex -c \
  "SELECT id, found_url, alert_type, created_at FROM alerts WHERE alert_type = 'instagram_match' ORDER BY created_at DESC LIMIT 5;"

Verificar perfis auto-descobertos (paginado na API, direto no DB):

bash
docker compose exec -T db psql -U kubex -d kubex -c \
  "SELECT username, source, discovered_via, confidence, user_id FROM monitored_instagram_profiles WHERE source = 'auto' ORDER BY created_at DESC LIMIT 10;"

Verificar via API (paginada):

bash
curl -s "http://admin.kubex.test/api/v1/admin/social/instagram/profiles?source=auto&page=1&page_size=10" \
  --cookie "access_token=..." | python -m json.tool

Nota: Scan a cada 3h (max 20 posts/perfil). Discovery: Beat 6d, cooldown 7d por user (max 10 variantes, 5 perfis/variante, 100 perfis cap). FK user_id ON DELETE SET NULL — perfis sobrevivem exclusao de conta. Risco de ban de IP — monitorar circuit breaker.

FaceID — Reconhecimento Facial (InsightFace + Milvus) ​

O que faz: Detecta rostos com InsightFace AntelopeV2 (512D) e compara contra embeddings indexados no Milvus via HNSW COSINE. Mock mode: Nao tem — requer InsightFace e Milvus rodando.

Para ativar:

  1. Verificar que "faceid" esta no search_engines

  2. Verificar containers Milvus:

    bash
    docker compose ps milvus etcd minio

    Esperado: milvus, etcd, minio — todos "Up" ou "healthy"

  3. Verificar que InsightFace esta instalado no face-worker:

    bash
    docker compose exec face-worker python -c "import insightface; print('InsightFace OK')"
docker compose exec face-worker python -c "from pymilvus import connections; connections.connect(); print('Milvus OK')"

  4. Verificar variaveis no .env:

    env
    MILVUS_HOST=milvus
MILVUS_PORT=19530
INSIGHTFACE_DET_THRESH=0.5
INSIGHTFACE_GPU=true

  5. O usuario precisa cadastrar selfie (identidade) pelo frontend

  6. GPU (CUDA) opcional — CPU suficiente para dev

Testar endpoint /face/detect:

bash
# Requer imagem com rosto
curl -X POST http://kubex.test/api/v1/face/detect \
  -H "Cookie: access_token=<TOKEN>" \
  -F "file=@foto_com_rosto.jpg"

Esperado: {"faces_detected": 1, "bounding_boxes": [[...]], "sha1": "..."}

Testar endpoint /face/compare:

bash
curl -X POST http://kubex.test/api/v1/face/compare \
  -H "Cookie: access_token=<TOKEN>" \
  -F "file=@foto_com_rosto.jpg"

Verificar colecao Milvus:

bash
docker compose exec face-worker python -c "
from pymilvus import connections, utility
connections.connect(host='milvus', port=19530)
print('Colecoes:', utility.list_collections())
"

Lens — DDG Fallback ​

Sentinel — Self-Healing Watchdog ​

O que faz: Motor autonomo de auto-recuperacao que roda a cada 5 minutos via Celery Beat. Executa ciclo de 3 fases:

  1. Diagnose — Verifica saude dos motores
  2. Heal — Aplica estrategias de recuperacao automatica (reset CB, retry DLQ, revoke zombies)
  3. Report — Registra issues e metricas de recuperacao

Mock mode: Nao tem — sempre roda. Em dev, logs debug sem crash se servicos indisponiveis.

Estrategias de recuperacao:

  • Circuit breakers stuck em OPEN → force reset
  • Motores indisponiveis → force CB OPEN (economizar recursos)
  • DLQ retry com backoff exponencial (30min × 2^attempts, max 3 tentativas)
  • Deteccao de error trends (alerta >80%, escala >50%)
  • Queue backlog (>50 tasks reservadas)
  • Zombie tasks (>10min) → revoke

Nenhuma configuracao necessaria. Ativo automaticamente com Celery Beat.

Testar manualmente:

bash
docker compose exec -T celery-worker celery -A app.tasks call sentinel.run_watchdog_cycle

docker compose logs celery-worker --tail=30 | grep -i sentinel

Esperado: log mostrando "Ciclo concluido" com issues encontradas e acoes tomadas

Verificar historico no Redis:

bash
docker compose exec redis redis-cli -a kubex_dev_pass LRANGE kubex:sentinel:history 0 5
docker compose exec redis redis-cli -a kubex_dev_pass HGETALL kubex:sentinel:stats
