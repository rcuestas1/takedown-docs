Usuário Read-Only do PostgreSQL para Agentes IA — Kubex ​

Versão: 1.0 Data: 2026-03-27

1. Motivação ​

Os agentes IA do KAAS (Kubex Agent Autonomy System) possuem a ferramenta query_db_readonly que executa consultas SQL no banco de dados. Embora a ferramenta valide que apenas comandos SELECT sejam executados, essa validação é feita em nível de aplicação — um bypass de parsing (por exemplo, via subqueries, CTEs ou ofuscação de keywords) poderia permitir mutações no banco.

Defesa em profundidade: criar um usuário PostgreSQL com permissões exclusivamente de leitura garante que, mesmo que a validação da aplicação seja contornada, o banco de dados rejeitará qualquer tentativa de INSERT, UPDATE, DELETE, DROP, TRUNCATE ou ALTER.

Riscos mitigados ​

  • SQL injection via agente IA: prompt injection que leve o agente a executar queries maliciosas
  • Bug no parser de validação: bypass da checagem startswith("SELECT") em _exec_db_query
  • Escalação de privilégios: agente comprometido não consegue alterar dados

2. Criação do Usuário Read-Only ​

Execute os comandos abaixo no PostgreSQL de produção (e opcionalmente em dev):

sql
-- Criar usuario read-only para agentes IA
CREATE USER kubex_agent_ro WITH PASSWORD 'CHANGE_ME_USE_STRONG_PASSWORD';

-- Permitir conexao ao banco
GRANT CONNECT ON DATABASE kubex TO kubex_agent_ro;

-- Permitir uso do schema public
GRANT USAGE ON SCHEMA public TO kubex_agent_ro;

-- Conceder SELECT em todas as tabelas existentes
GRANT SELECT ON ALL TABLES IN SCHEMA public TO kubex_agent_ro;

-- Garantir que tabelas futuras tambem tenham SELECT
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO kubex_agent_ro;

Verificação ​

sql
-- Testar que o usuario nao consegue fazer mutacoes
SET ROLE kubex_agent_ro;

-- Deve funcionar:
SELECT COUNT(*) FROM users;

-- Deve falhar com "permission denied":
DELETE FROM users WHERE id = 1;
UPDATE users SET is_admin = true WHERE id = 1;

RESET ROLE;

Via Docker (ambiente dev) ​

bash
# Acessar o container do PostgreSQL
docker exec -it kubex-db psql -U kubex -d kubex

# Executar os comandos SQL acima

3. Configuração no .env ​

Adicionar uma nova variável de ambiente para a conexão read-only:

env
# Conexao principal (leitura + escrita) — usada pela aplicacao
DATABASE_URL=postgresql+asyncpg://kubex:senha@db:5432/kubex

# Conexao read-only — usada exclusivamente pelos agentes IA
AGENT_DATABASE_URL=postgresql+asyncpg://kubex_agent_ro:CHANGE_ME@db:5432/kubex

Em app/config.py ​

Adicionar o campo ao Settings:

python
AGENT_DATABASE_URL: str | None = None  # Read-only connection for AI agents

4. Alteração em app/agents/tools.py ​

A função _exec_db_query em app/agents/tools.py deve ser atualizada para usar a conexão read-only quando disponível:

python
async def _exec_db_query(query: str) -> str:
    # SECURITY: Use read-only database connection (AGENT_DATABASE_URL)
    # to prevent SQL injection via AI agent tools.
    from app.config import settings

    if settings.AGENT_DATABASE_URL:
        # Usar conexao read-only dedicada
        engine = create_async_engine(settings.AGENT_DATABASE_URL)
        async with AsyncSession(engine) as session:
            result = await asyncio.wait_for(session.execute(sa_text(query)), timeout=30)
            # ... processar resultado
    else:
        # Fallback: usar conexao principal (menos seguro)
        from app.database import async_session
        async with async_session() as session:
            # ... codigo atual

Nota

A rotacao de credenciais para o usuario read-only e configuravel via variavel de ambiente. Consulte app/agents/tools.py, funcao _exec_db_query.

5. Tabelas com Dados Sensíveis ​

O usuário read-only terá acesso de leitura a todas as tabelas. Para maior segurança, considerar revogar acesso a tabelas específicas:

sql
-- Revogar acesso a tabelas com dados sensiveis (opcional)
REVOKE SELECT ON users FROM kubex_agent_ro;  -- contem password_hash
REVOKE SELECT ON audit_logs FROM kubex_agent_ro;  -- contem IPs

-- Criar views filtradas se necessario
CREATE VIEW agent_users AS
SELECT id, public_id, email, is_active, is_admin, created_at
FROM users;

GRANT SELECT ON agent_users TO kubex_agent_ro;
