Plano de Resposta a Incidentes de Segurança — Kubex ​

Versão: 1.0 Última revisão: 2026-03-27 Responsável: Renato (CEO / DPO) Base legal: LGPD Art. 46, 48, 50

1. Objetivo ​

Este plano estabelece os procedimentos para detecção, contenção, erradicação e recuperação de incidentes de segurança da informação na plataforma Kubex, com foco especial na proteção de dados pessoais conforme exigido pela LGPD (Lei Geral de Proteção de Dados — Lei nº 13.709/2018).

A Kubex processa dados sensíveis incluindo:

  • Dados biométricos: embeddings faciais (Milvus), imagens para reconhecimento facial (AdaFace, DeepFace)
  • Dados pessoais: CPF/CNPJ, e-mail, nome, endereço IP
  • Dados de navegação: URLs monitoradas, resultados de varredura dos 27 motores de detecção
  • Credenciais: hashes bcrypt, tokens de sessão (httpOnly cookies)

2. Classificação de Incidentes ​

CRÍTICO (P1) — Resposta imediata (< 1 hora) ​

ExemploSistemas afetados
Vazamento de embeddings faciais (Milvus)Milvus, PostgreSQL, R2
Acesso não autorizado ao banco de dados de produçãoPostgreSQL, async_session
Comprometimento de credenciais de acesso (API keys, tokens)Redis, .env, Celery broker
Ransomware ou destruição de dadosTodos os sistemas
Exploração ativa de vulnerabilidade (RCE, SQL injection)FastAPI, PostgreSQL
Vazamento massivo de PII (CPF, CNPJ, dados de usuários)PostgreSQL, audit_logs

ALTO (P2) — Resposta em até 4 horas ​

ExemploSistemas afetados
Acesso não autorizado a painel administrativofrontend-admin, auth cookies
Bypass de autenticação ou escalação de privilégiosFastAPI auth, get_admin_user
Falha no circuit breaker expondo APIs externasapp/core/circuit_breaker.py
Comprometimento de worker CeleryCelery, Redis broker
SSRF bypass no crawler engineapp/engines/crawler/
Exposição de dados via API sem autenticaçãoFastAPI routes

MÉDIO (P3) — Resposta em até 24 horas ​

ExemploSistemas afetados
Falha persistente em motor de detecção (health check)Motor Registry, Sentinel
Rate limiting ineficaz (DDoS parcial)slowapi, Redis, Nginx
Logs contendo PII indevidamentelogging, audit_logs
Certificado SSL próximo ao vencimentoNginx, kubex.com.br
Vulnerabilidade em dependência (pip-audit/npm audit)requirements.txt, package.json

BAIXO (P4) — Resposta em até 72 horas ​

ExemploSistemas afetados
Tentativas de login brute-force (bloqueadas)slowapi, audit_logs
Scan de portas detectadoNginx, Docker network
Falha intermitente em mock modeTelegram mock, engines
Warning de segurança em header CSPNginx security headers

Escalonamento ​

Agente detecta problema → Director Agent avalia severidade
  → P3/P4: Agente resolve e reporta
  → P1/P2: Notifica CEO IMEDIATAMENTE + Agente inicia contenção

4. Procedimento de Resposta ​

4.1 Detecção e Triagem (todas as severidades) ​

  1. Identificar a fonte: log de auditoria, alerta do Sentinel (Motor 27), health check, pip-audit, monitoramento externo
  2. Classificar a severidade conforme Seção 2
  3. Registrar o incidente:
    • Data/hora de detecção
    • Descrição técnica
    • Sistemas afetados
    • Dados pessoais potencialmente expostos
    • Severidade atribuída
  4. Notificar a equipe conforme a severidade

4.2 Contencao ​

P1 (CRITICO) ​

  1. Isolar servico comprometido: docker stop kubex-api (ou container afetado)
  2. Se Milvus comprometido: docker stop kubex-milvus
  3. Revogar credenciais — rotacionar SECRET_KEY, DATABASE_URL, REDIS_URL em .env
  4. Bloquear IPs suspeitos via nginx/default.conf
  5. Preservar evidencias: docker logs kubex-api > /tmp/incident_$(date +%Y%m%d_%H%M%S).log

P2 (ALTO) ​

  1. Invalidar sessoes: limpar tokens no Redis (KEYS "kubex:session:*")
  2. Ativar rate limiting agressivo temporariamente
  3. Verificar audit_logs da ultima hora

P3/P4 (MEDIO/BAIXO) ​

  1. Documentar o problema
  2. Aplicar correcao ou workaround
  3. Agendar correcao definitiva

4.3 Erradicação ​

  1. Identificar a causa raiz — análise de logs, code review, scan de vulnerabilidades
  2. Remover o vetor de ataque:
    • Patch de código (branch kaas/security/incident-{id})
    • Atualização de dependências vulneráveis
    • Correção de configuração
  3. Verificar que não há persistência:
    • Verificar crontabs e scheduled tasks
    • Verificar se não há backdoor em código
    • Scan com pip-audit e npm audit
  4. Rotacionar TODAS as credenciais se houve comprometimento de acesso

4.4 Recuperação ​

  1. Restaurar serviços na ordem:
    • PostgreSQL → Redis → FastAPI (api) → Celery workers → Nginx → Frontends
  2. Verificar integridade:
    bash
    # Health check dos motores
python -m pytest tests/ -m "not slow"
python -m ruff check app/

# Verificar Sentinel
curl http://localhost:8000/api/v1/sentinel/health
  3. Monitorar por 24-48 horas após restauração
  4. Validar que dados LGPD não persistiram indevidamente:
    • Confirmar del img_bytes em todas as tasks
    • Verificar que não há PII em logs

4.5 Comunicação ​

Comunicação Interna ​

  • P1/P2: CEO notificado imediatamente (WhatsApp, e-mail)
  • P3/P4: Relatório no próximo ciclo de revisão

Comunicação Externa (ver Seção 5 para prazos LGPD) ​

  • ANPD: obrigatório para incidentes envolvendo dados pessoais
  • Titulares: obrigatório quando há risco relevante aos direitos

5. Prazos LGPD (Art. 48) ​

Notificação à ANPD ​

A LGPD (Art. 48, §1º) determina que o controlador deve comunicar à Autoridade Nacional de Proteção de Dados (ANPD) e ao titular a ocorrência de incidente de segurança que possa acarretar risco ou dano relevante aos titulares.

Prazo: A ANPD recomenda comunicação em até 2 dias úteis a partir do conhecimento do incidente (Resolução CD/ANPD nº 15/2024).

Quando notificar a ANPD ​

Notificar OBRIGATORIAMENTE quando o incidente envolver:

  • Dados pessoais sensíveis (biométricos — embeddings faciais)
  • Dados de crianças/adolescentes
  • Dados que possam causar dano material ou moral significativo
  • Grande volume de titulares afetados

Conteúdo da notificação (Art. 48, §1º) ​

A comunicação deve conter, no mínimo:

  1. Descrição da natureza dos dados pessoais afetados
  2. Informações sobre os titulares envolvidos
  3. Indicação das medidas técnicas e de segurança utilizadas
  4. Riscos relacionados ao incidente
  5. Motivos da demora, se não comunicado imediatamente
  6. Medidas adotadas para reverter ou mitigar os efeitos

Notificação aos Titulares ​

Quando o incidente puder acarretar risco ou dano relevante, os titulares devem ser notificados com:

  • Descrição clara e acessível do ocorrido
  • Tipos de dados afetados
  • Medidas tomadas
  • Recomendações ao titular (troca de senha, monitoramento)
  • Canal de contato do DPO/Encarregado

6. Templates de Notificacao ​

6.1 Template — Notificacao a ANPD ​

Conteudo minimo (Art. 48, par.1):

  1. Identificacao do controlador — Razao social, CNPJ, DPO (Renato)
  2. Descricao do incidente — Data, tipo (acesso nao autorizado / vazamento / perda)
  3. Dados pessoais afetados — Tipos (nome, CPF, biometricos), volume de titulares
  4. Medidas de seguranca pre-existentes — httpOnly cookies, bcrypt, rate limiting, SSRF guard, LGPD (del img_bytes), circuit breakers, audit logs
  5. Riscos aos titulares
  6. Medidas adotadas para contencao/mitigacao
  7. Assinatura — Renato (DPO / CEO), data

6.2 Template — Notificacao aos Titulares ​

Conteudo minimo:

  1. O que aconteceu — Descricao em linguagem acessivel
  2. Dados afetados — Lista dos tipos de dados
  3. Acoes tomadas — Medidas tecnicas e organizacionais
  4. Recomendacoes — Alterar senha, monitorar contas, desconfiar de comunicacoes suspeitas
  5. Canal de contato — Email e telefone do DPO

8. Pós-Incidente ​

Análise Post-Mortem ​

Após a resolução de qualquer incidente P1 ou P2, realizar análise post-mortem contendo:

  1. Timeline completa — desde a primeira evidência até a resolução
  2. Causa raiz — análise técnica detalhada
  3. Impacto real — dados afetados, titulares impactados, duração
  4. Eficácia da resposta — o que funcionou, o que precisa melhorar
  5. Ações corretivas — lista de melhorias com prazos e responsáveis
