Snapshot do Banco de Dados ​

Pagina do painel admin que exibe estatisticas consolidadas do banco de dados PostgreSQL e operacoes dos motores de busca.

URL: /database no painel admin (admin.kubex.test/database)

Visao Geral ​

A pagina combina dados de 4 endpoints para criar uma visao unificada:

EndpointDadosFonte
GET /admin/database/snapshotTotais do banco + contagem por tabelaPostgreSQL catalog
GET /admin/metrics/motorsScans, matches, errors por motorRedis
GET /admin/activity-logs/statsResumo operacoes 24hPostgreSQL
GET /admin/activity-logs?level=ERRORUltimos 10 errosPostgreSQL

Secoes da Pagina ​

1. KPIs do Banco de Dados ​

4 cards com totais do PostgreSQL:

KPIDescricaoFonte SQL
Total de RegistrosSoma de registros em todas as tabelaspg_stat_user_tables.n_live_tup ou COUNT(*)
TabelasNumero de tabelas no schema publicinformation_schema.tables
IndicesNumero de indicespg_indexes
Relacionamentos (FK)Numero de foreign keysinformation_schema.table_constraints

2. KPIs dos Motores ​

4 cards com resumo de operacoes:

KPIDescricaoFonte
Operacoes 24hTotal de logs de atividade nas ultimas 24hactivity-logs/stats
Scans HojeTotal de execucoes de todos os motores hojemetrics/motors (Redis)
Matches HojeTotal de correspondencias encontradas hojemetrics/motors (Redis)
Erros 24hTotal de erros nas ultimas 24hactivity-logs/stats

Quando ha erros, um alerta vermelho lista quais motores apresentaram falhas.

3. Registros por Tabela ​

Grafico de barras horizontais mostrando quantidade de registros em cada tabela, ordenado por volume. Largura da barra proporcional ao maior valor.

4. Scans por Motor ​

Grafico de barras horizontais com total de scans por motor. Cada linha mostra:

  • Nome legivel do motor (ex: "Crawler — Web Crawler")
  • Barra proporcional de scans
  • Contagem de matches (verde) e erros (vermelho)

5. Ultimos Erros ​

Tabela com os 10 erros mais recentes:

  • Quando — timestamp
  • Motor — qual motor falhou
  • Mensagem — descricao do erro (truncada)
  • Trace — link para timeline completa no Activity Logs

Link "Ver todos" navega para /activity-logs?level=ERROR.

Controles ​

Toggle Estimado / Exato ​

  • Estimado (default): usa pg_stat_user_tables.n_live_tup — rapido (~10ms), precisao depende do autovacuum
  • Exato: usa COUNT(*) real via UNION ALL — preciso, mais lento (~500ms com 30 tabelas)

Dica: A estimativa e suficiente para monitoramento diario. Use modo exato quando precisar de numeros precisos para relatorios.

Botao Atualizar ​

Re-executa todas as 4 chamadas de API simultaneamente.

API: GET /admin/database/snapshot ​

Request ​

bash
curl http://kubex.test/api/v1/admin/database/snapshot?exact=false \
  -b cookies.txt
ParamTipoDefaultDescricao
exactboolfalseContagem exata (COUNT(*)) ou estimativa (pg_stat)

Response (200) ​

json
{
  "snapshot_at": "2026-03-15T14:30:00Z",
  "totals": {
    "total_records": 125430,
    "total_tables": 30,
    "total_indexes": 67,
    "total_relationships": 42
  },
  "tables": [
    {"name": "audit_logs", "rows": 45000},
    {"name": "search_results", "rows": 32000},
    {"name": "alerts", "rows": 12500},
    {"name": "engine_activity_logs", "rows": 8200},
    {"name": "content_hashes", "rows": 5100}
  ],
  "exact": false
}

Response (500) ​

json
{
  "detail": "Erro ao consultar estatisticas do banco de dados"
}

Seguranca ​

  • Requer is_admin=True (cookie httpOnly)
  • Rate limit: 30 req/min por IP
  • Audit log registrado: acao admin_database_snapshot
  • Nomes de tabela validados contra regex ^[a-z_][a-z0-9_]{0,62}$
  • Tabelas vem exclusivamente do catalogo PostgreSQL (nunca de input do usuario)
  • Query de contagem exata usa UNION ALL unico (sem N+1)
