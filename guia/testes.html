Estrategia de Testes ​

O Kubex usa pytest com fixtures extensivas para testar API, engines, core modules e middleware.

Executar Testes ​

bash
# Todos os testes
docker compose exec api pytest --tb=short -q

# Com cobertura
docker compose exec api pytest --cov=app --cov-report=term-missing

# Um arquivo especifico
docker compose exec api pytest tests/test_api_auth.py -v

# Um teste especifico
docker compose exec api pytest tests/test_api_auth.py::test_login_success -v

# Apenas testes rapidos (sem banco)
docker compose exec api pytest -m "not slow" -q

Estrutura ​

tests/
├── conftest.py                    # Fixtures globais (users, clients, mocks)
├── test_api_*.py                  # Testes de endpoints API (17 arquivos)
├── test_core_*.py                 # Testes de modulos core (7 arquivos)
├── test_engines_*.py              # Testes de engines/monitors (8 arquivos)
├── test_middleware_*.py           # Testes de middleware (2 arquivos)
├── test_celery_*.py               # Testes de tasks Celery (4 arquivos)
├── test_models_smoke.py           # Smoke test de todos os models
└── test_routers_smoke.py          # Smoke test de todos os routers

Usuarios ​

Clientes HTTP ​

FixtureAuthUso

Mocks Automaticos (autouse) ​

  • Rate limiter — desabilitado em todos os testes
  • Maintenance mode — desabilitado em todos os testes
  • Celery — tasks executam sync (sem broker)
  • Database — usa FakeAsyncSession (in-memory)
  • Redis — usa FakeRedis (in-memory)

CI ​

O CI exige cobertura minima de 60%:

bash
pytest --cov=app --cov-report=term-missing --cov-fail-under=60

Ver CI/CD para detalhes do pipeline.

Convencoes ​

  1. Naming: test_<modulo>.py → test_<funcao> ou test_<cenario>
  2. Fixtures: Sempre usar fixtures do conftest.py em vez de criar mocks locais
  3. Async: Testes de endpoint usam httpx.AsyncClient via client fixture
  4. Isolamento: Cada teste e independente — sem estado compartilhado entre testes
  5. Mock mode: Engines usam mock automaticamente quando API key esta vazia
